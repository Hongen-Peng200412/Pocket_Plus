这份文件的定位
=========================================================================================================================
这是一个AI的片面建议，不要作为指导手册









本次修改的出发点
==================

当前训练与评估主要还是 voxel 语义分割范式，最后还需要再把 voxel 概率映射回 atom 概率，这里天然存在 gap。我的目标不是立刻改第二阶段，而是先把第一阶段改成一个更适合“atom-level pocket semantic segmentation”的模型：让 CNN 专门读取 BOX 内的连续密度信息，让点云网络专门读取 BOX 内/邻域原子的结构信息，再把二者在点上融合，最终直接输出原子是否属于 pocket 的概率。

本次首版总体思路
================

1. 第一阶段不再是纯 voxel 分割，而是改成 voxel branch + point branch 并行。
2. voxel branch 负责处理 BOX 的密度类输入，包括真实密度图、模拟密度图、差图、以及现有的体素化受体特征。
3. point branch 首版只处理 BOX 内或 BOX 邻域的真实原子，不处理整个 PDB。
4. point branch 使用轻量化的 PTV3 encoder，而不是直接照搬完整 UniPocket 流程。
5. 只做单向 voxel -> point 融合，不做 point -> voxel 回写。
6. 融合次数控制在 1 到 2 次，中间最多一次，末端固定一次。
7. 最终在点上拼接 voxel 与 point 特征，再接一个轻量点云 transformer 分类头，直接输出 atom-level logits。
8. 首版不引入伪原子；伪原子只作为后续增强选项。
9. 首版保留一个轻量 voxel auxiliary head，避免第一阶段 recall 明显下降。

首版边界与不做的事
==================

1. 不做 whole-PDB point branch。首版只看 BOX 内/邻域原子。
2. 不做 diffusion，不做第二阶段改造。
3. 不做 point <-> voxel 双向多次交换。
4. 不做复杂虚拟节点系统。若后续需要伪原子，也只在点分支内部加入。
5. 不追求第一次就更换全部数据分辨率。先优先复用现有 BOX 数据流跑通；之后再考虑 1.5A 版本。

总实施路线
==========

Phase 0. 先复用现有资源，明确数据来源
-----------------------------------

需要同时利用以下几类已有资源：

1. BOX 级体素数据：来自 Pocket++/processedPDB_EMDB_binder/split_and_select_box.py 切出的 emdb_BOX / pdb_feature_BOX / pdb_label_BOX。
2. 原子级数据：来自 Pocket++/Make_Data/process_and_label.py 产出的 atoms.npz / labels.npz / graph.npz。
3. 点云 backbone 参考：来自 C:/Users/15919/OneDrive/My_Project/UniPocket/model.py 的 PointTransformerV3 与相关适配结构。

本阶段要确认的事情：

1. 当前 BOX .npz 内是否保存了 grid / origin / voxel_size / x_range / y_range / z_range。答案是是的，split_and_select_box.py 已经保存了这些字段，因此可以从 BOX 反查空间范围。
2. 当前 sample_name 中是否包含 pdb_id。答案是是的，文件名形如 {pdb_id}_{instance_id}_{Rxx}_{Ryy}_{Rzz}[_C].npz，可以据此回到原始样本目录读取 atoms.npz 与 labels.npz。
3. 当前训练 Dataset 只返回 grid / label / hardmask，不包含 atom_coords / atom_labels，因此必须新增一套 dataset，而不是在现有 MyDatasets 上小修小补。

Phase 1. 先建立新的 box+point 数据流
----------------------------------

目标：让每个训练样本不再只是一个体素张量，而是一个同时包含 voxel 与 atom 信息的 dict。

建议新增文件：

1. src/datasets/box_point_dataset.py
2. src/datasets/box_point_collate.py

建议新增数据样本结构：

1. voxel_grid: torch.float32, (C, D, H, W)
2. voxel_label: torch.int64, (D, H, W)
3. hardmask: torch.int64, (D, H, W)
4. box_origin_world: float, (3,)
5. voxel_size_world: float, (3,)
6. atom_coord_world: float, (N, 3)
7. atom_coord_local: float, (N, 3)
8. atom_feat: float, (N, F_atom)
9. atom_label: long/bool, (N,)
10. atom_valid_mask: bool, (N,)
11. atom_is_in_core_box: bool, (N,)
12. sample_name / pdb_id / class_name / instance_id 等元信息

其中 atom_coord_local 的定义建议固定为：

1. 以 BOX 自身 origin 为参考，先把世界坐标转为 local world 坐标。
2. 再根据 voxel_size 换算成连续 voxel 坐标，供 trilinear sampling 使用。
3. 同时保留一个以 BOX 中心为原点、单位为 Angstrom 的相对坐标版本，供 point 分支和分类头使用。

BOX 内原子筛选规则建议：

1. 首版只取 BOX 内原子与 BOX 外扩 r_buffer 的邻域原子。
2. r_buffer 初始建议 3.0A 或 4.0A，不要太大。
3. 若原子数超过上限 N_max，则优先保留 core box 内原子，再按距 BOX 中心或距配体中心的距离裁剪 buffer 原子。

atom_label 的定义建议：

1. 二分类首版直接用 labels.npz 的 binding_mask。
2. 如果后续要做多分类，再切换为 pocket_class_ids。
3. 首版只对真实原子计算主损失，不对未来伪原子计算监督。

atom_feat 的首版建议：

1. 直接复用 atoms.npz 中的 49 维原子特征。
2. 额外拼接 atom_is_in_core_box。
3. 额外拼接 atom 相对 BOX 中心的位置编码，如 rel_xyz / r / r^2。

数据增强必须同步修改：

1. 当前 random_rotation90_plus 只处理 voxel 与 hardmask，不会同步旋转 atom 坐标。
2. 新 dataset 必须实现“voxel、label、hardmask、atom_coord_local”同步旋转。
3. 旋转轴与旋转中心必须严格与 voxel 增强保持一致，否则 voxel->point 融合会系统错位。
4. 若未来加入伪原子，也必须参加同样的几何变换。

为了减少每个 epoch 的磁盘随机读取，建议增加缓存策略：

1. 首选每个 pdb_id 的 atoms.npz / labels.npz 做内存级 LRU cache。
2. 如果训练 I/O 仍然慢，再考虑预生成“BOX 对应 atom index 列表”的中间缓存文件。

Phase 2. 新建 voxel+point backbone
---------------------------------

目标：不破坏原有 RAUNet，而是新增一个专门的 stage1 backbone。

建议新增文件：

1. src/model/voxel_point_stage1.py
2. src/model/point_backbones/ptv3_light.py
3. 如有必要，新增 src/model/fusion.py

总体结构：

1. voxel branch: 基于现有 RAUNet/SimpleUnet 改造，输出中间 voxel feature 和末端 voxel feature。
2. point branch: 轻量 PTV3 encoder，只处理该 BOX 的 atom set。
3. fusion1: 中层 voxel -> point
4. fusion2: 末层 voxel -> point
5. point head: 轻量 point transformer 分类头
6. voxel aux head: 低权重辅助监督

2.1 voxel branch 的具体建议
---------------------------

不要直接使用现有 SimpleUnet 的单一返回值，因为它只返回最终 logits。建议复制并改造为一个返回 dict 的 backbone。

建议返回这些中间特征：

1. voxel_feat_mid: 例如 c1 或 c0 前后的高分辨率特征
2. voxel_feat_final: 当前 conv_end 之前的 96 通道特征
3. voxel_logits_aux: 由 voxel_feat_final 经过一个轻量 head 得到的辅助 logits

建议原则：

1. 首版先保留现有 72^3 BOX 与当前数据通道配置。
2. 不在第一版里同时重做分辨率与网络结构，否则变量太多。
3. 如果后续验证 point branch 的确有效，再考虑把 density 通道改成 1.5A 版本并重切 BOX。

2.2 point branch 的具体建议
---------------------------

建议基于 UniPocket 的 PointTransformerV3 思路做一个裁剪版，而不是直接整文件复制使用。

原因：

1. UniPocket/model.py 依赖 spconv、torch_scatter、flash_attn，工程依赖较重。
2. 其中 flash_attn 相关导入需要做安全处理，不能假定本环境一定齐全。
3. 我们首版只需要 encoder，不需要完整 decoder 和其他不相关的任务头。

point branch 首版建议：

1. 只保留 encoder。
2. enc_depth、channels、num_heads 全部做轻量化。
3. 输入点特征 = atom_feat + rel_pos encoding + core/buffer flag。
4. 输入坐标 = atom_coord_local_centered 或其对应 grid_coord。
5. grid_size 与 point 下采样粒度独立于 voxel_size 配置，但初版不宜太细。

建议新增一个本地 wrapper，例如：

1. class PTV3AtomEncoder(nn.Module)
2. 负责把 batch 内所有 BOX 的 atom set 打包成 PointTransformerV3 所需的 data_dict
3. 输出每个 atom 对齐后的 point features

Phase 3. 设计 voxel -> point 融合
--------------------------------

这是整个方案最关键的公共模块。

建议新增文件：

1. src/model/voxel_point_sampling.py
2. 或并入 src/model/fusion.py

首版只做单向 voxel -> point。

具体做法：

1. 以 atom_coord_local_voxel 为连续坐标。
2. 在 voxel feature map 上做 trilinear sampling，得到每个 atom 对应的 sampled voxel feature。
3. sampled feature 先经过一层线性投影或小 MLP。
4. 再与当前 point feature 做 concat 或 residual add。

首版推荐的融合方式：

1. fusion1 用 concat + linear，而不是直接相加。
2. fusion2 末层固定用 concat，因为最终我们本来就想把 voxel 与 point 信息都保留下来。

推荐的两个融合点：

1. fusion1: 在 voxel high-res feature 上采样，例如 c0 或者与 c0 同分辨率的特征。
2. fusion2: 在 voxel_feat_final 上采样，即 conv_end 前的 96 通道特征。

不建议首版做的事：

1. 不要每一层都融合。
2. 不要 point->voxel scatter 回写。
3. 不要多尺度全都上。

Phase 4. 末端点云 transformer 分类头
-----------------------------------

最终目标是输出 atom-level binary logits。

建议新增文件：

1. src/model/atom_heads.py

推荐结构：

1. 输入 = concat(point_feat_final, sampled_voxel_feat_final, rel_xyz, core/buffer flag)
2. 先做一层线性升维
3. 再做 2 到 3 层轻量 point transformer blocks
4. 最后用 MLP 输出每个 atom 的 1 维 logit

为什么末端还要再加一个点云 transformer：

1. PTV3 主干负责较强的点云编码，但经过两次 voxel 融合后，仍然需要一个专门面向“box 内 atom segmentation”的 head 做再整合。
2. 末端 head 可以更小、更贴任务，不必直接复用主干最后一层输出做分类。
3. 这一步也为后续伪原子/虚拟节点扩展留出接口。

首版 head 的设计要求：

1. 必须支持变长 atom 数。
2. 必须按 BOX 为单位隔离 attention，不要跨样本混 attention。
3. 若实现复杂度太高，第一版可以退化为 padding + attention mask 的标准 TransformerEncoder。
4. 如果 padding 方案显存过高，再切换为基于 offset 的局部 point transformer blocks。

Phase 5. voxel auxiliary head 与损失设计
---------------------------------------

首版不建议彻底抛弃 voxel 监督。

原因：

1. 当前 baseline 已经有较高 recall，不能让第一阶段直接丢掉这个优势。
2. voxel aux head 可以让 voxel branch 继续学习密度语义，而不只是充当 point branch 的前端。

建议损失：

1. 主损失：atom-level binary focal 或 BCE/focal
2. 辅损失：现有 voxel loss，继续使用 hardmask
3. 总损失：loss = loss_atom + lambda_voxel * loss_voxel

lambda_voxel 首版建议：

1. 先设较小，如 0.2 或 0.3
2. 先保证 atom head 学得起来
3. 如果 recall 明显下降，再提高 voxel 辅助权重

atom-level 评价指标建议：

1. atom PR-AUC
2. atom precision / recall
3. core box 内原子的 precision / recall
4. box-level 是否命中 pocket 的 hit rate

如果旧流程仍然需要 voxel probability：

1. 暂时保留 voxel aux head 供兼容推断
2. 或在推断后把 atom logits scatter 回局部 voxel，但这不作为首版主路径

Phase 6. 新 wrapper、新 datamodule 兼容与训练入口
-----------------------------------------------

当前 VolumeSegmentationWrapper 与 train.py 假设 batch 的第一个元素是纯 voxel tensor，因此本方案需要新 wrapper。

建议新增文件：

1. src/wrappers/voxel_point_stage1.py

建议职责：

1. 接收 dict batch，而不是 (x, y, hardmask)
2. forward 传入 voxel + point + meta
3. training_step 计算 atom 主损失和 voxel 辅助损失
4. validation_step 记录 atom 级别指标
5. 保留 optimizer / scheduler 接口风格，与当前 Hydra 体系兼容

train.py 需要做的兼容性调整：

1. _extract_input_tensor 不能再只识别 tuple[0] 为 tensor 的情况
2. lazy init 需要能从 dict batch 中取出 voxel_grid
3. dataloader 需要支持自定义 collate_fn
4. 如果 point batch 不是 PyG Data，就继续使用 torch DataLoader 即可

Phase 8. 伪原子/虚拟点的后续扩展
--------------------------------

这不是首版必做项，但必须预留接口。

未来如果要加伪原子，建议原则如下：

1. 伪原子只加在 point branch，不回写 voxel。
2. 伪原子的初始坐标优先来自 BOX 中心附近、差图显著区域、或规则采样点。
3. 伪原子的初始特征主要来自 voxel_feat 的局部采样，而不是手写化学特征。
4. 伪原子参与 attention 和特征传播，但默认不参与监督损失。
5. 真正要监督时，也应先做 auxiliary objective，而不是直接把伪原子当真实原子标签训练。

建议的首版里程碑
================

Milestone 1. 数据先跑通
----------------------

1. 新 dataset 能从一个 BOX 样本正确取到对应 atoms.npz / labels.npz。
2. 能正确筛出 BOX 内/邻域原子。
3. 能正确输出 atom_coord_local、atom_feat、atom_label。
4. 数据增强后 voxel 与 atom 坐标仍然严格对齐。

Milestone 2. 先做最小闭环，不接 PTV3
------------------------------------

1. voxel branch 先用现有 RAUNet 改造成返回中间特征
2. voxel_feat 直接采样到 atom
3. sampled voxel feat + atom_feat -> 小 MLP/小 transformer
4. 先证明 voxel->point 这条链路是通的

这样做的意义：

1. 先验证数据流与坐标系统
2. 先验证 atom-level supervision 是否稳定
3. 避免一上来就把所有不确定性都归因给 PTV3

Milestone 3. 接入轻量 PTV3 encoder
---------------------------------

1. 引入 point branch
2. 先只做末端一次融合
3. 如果稳定，再加中层 fusion1

Milestone 4. 再决定是否重做 1.5A 版本
------------------------------------

1. 若当前 72^3 数据下方案已经有效，再考虑重新生成更适合 density branch 的输入
2. 若直接改 1.5A，则必须同步检查 BOX 空间范围、sample naming、origin/voxel_size 与现有 labels 的兼容性

首版具体建议采用的实现顺序
==========================

1. 新建 box_point_dataset.py 与 collate_fn
2. 新建 voxel_point_stage1.py 的最小版本，只保留 voxel->point + 小头
3. 新建 wrapper，打通训练
4. 先做 atom-only 主指标验证
5. 再接入 ptv3_light.py
6. 再加第二个融合点
7. 最后再考虑伪原子与 1.5A 输入

关键注意事项
============

1. 坐标系统一定要统一：world coord、local coord、continuous voxel coord、PTV3 grid_coord 必须严格区分。
2. 数据增强一定要同步到 atom。只转 voxel 不转 atom 会直接把整个方案做坏。
3. 首版不要让 point 数过大，否则 PTV3 和末端 transformer 的显存都会不稳定。
4. 首版不要同时引入 whole-PDB、伪原子、双向融合、1.5A 重切盒子，这样实验不可解释。
5. 要保留清晰的消融顺序：
   - voxel baseline
   - voxel->point without PTV3
   - voxel + PTV3 with final fusion only
   - voxel + PTV3 with 2 fusions
   - + pseudo atoms
6. UniPocket 的 PTV3 代码只能作为资源，不应不加筛选地整个复制进主训练链路。
7. 当前 old wrapper 与 train.py 偏向纯 voxel 任务，所以这次改动更像新增一条 stage1 训练路线，而不是小修现有 VolumeSegmentationWrapper。

最终首版预期
============

若按以上顺序推进，首版最现实的目标不是立刻把整个项目精度大幅拉升，而是：

1. 把第一阶段输出从“voxel 概率图”更自然地转向“atom-level pocket 概率”
2. 缩小 voxel -> atom 的语义 gap
3. 为后续第二阶段的 CNN / 图分支 / 虚拟节点方案提供更干净的 atom-level先验
4. 在不显著损伤 recall 的前提下，改善第一阶段 proposal 的质量
