# YAML 配置注释规范

## Overview

这个 skill 用于为 Hydra/YAML 配置文件补充高信息密度的中文注释，风格参考当前仓库里已有的模型配置与数据集配置。

目标不是复述 YAML 语法，而是让读者快速看懂：

- 这个配置文件整体是给谁消费的。
- 某个 section 在训练或推理流程中扮演什么角色。
- 某个键的类型、意义、推荐取值、回退逻辑和运行时影响是什么。
- 某条 `# →` 注释指向的代码消费链是否已经被验证。

## 核心规则

- 使用中文注释解释配置含义，但保留英文技术标识原样，例如 `_target_`、`cfg.model.*`、`cfg.dataset.*`、类名、模块名、文件名、函数名、路径、JSON、Hydra。
- 注释优先说明“这个字段控制什么”和“它最终在哪里生效”，而不是机械重复键名。
- 只写当前上下文能够确认的真实行为；不要猜测不存在的默认值、调用链或代码位置。
- 允许同时存在规范型注释与实验型注释。规范型注释负责解释通用语义；实验型注释可以使用 `NOTE:` 说明当前实验意图。
- 对重要字段保持较高注释密度；对明显自解释的字段可以减少注释，避免把每一行都注释成自然语言复述。

## 注释层级

为 YAML 配置写注释时，优先按下面 5 个层级组织，而不是只在行尾零散补一句话。

### 1. 文件级横幅

在文件开头使用横幅注释，总结：

- 文件主题，例如模型配置、数据集配置、训练配置。
- 该文件通常以 `cfg.xxx.*` 的形式被谁读取。
- 该文件的主要消费者，例如 `train.py`、`Wrapper`、`DataModule`。

推荐格式：

```yaml
# ============================================================
# 某模型配置
# 此文件的所有配置项以 cfg.model.* 的形式被 train.py 和 Wrapper 使用
# ============================================================
```

### 2. `_target_` 与实例化入口注释

如果配置里包含 `_target_`，优先在它前面说明：

- `_target_` 的类型和语义。
- Hydra 如何利用它实例化对象。
- 该对象由哪个上游入口传入，例如 `train.py` 中的 `hydra.utils.instantiate(...)`。

推荐格式：

```yaml
# _target_: str, 目标类全称; Hydra 据此实例化 Wrapper
#   → train.py: hydra.utils.instantiate(cfg.model, ...)
_target_: src.wrappers.example.ExampleWrapper
```

### 3. section 级横幅

每个大块配置前使用：

```yaml
# --- 中文名 (English Alias) ---
```

section 横幅用于显式分组，例如：

- `# --- 骨干网络 (Backbone) ---`
- `# --- 损失函数 (Loss) ---`
- `# --- 数据根目录路径 (允许三个阶段分别指定) ---`
- `# --- 学习率调度器行为控制 ---`

不要只写抽象标题。标题应尽量让读者一眼知道本组字段的职责。

### 4. 键值同行注释

对标量键、布尔键、枚举键、短列表键，优先写成：

```yaml
key: value  # type, 意义, 必要时补充范围/单位/模式
```

推荐写入的信息优先级：

- 类型，例如 `int`、`float`、`bool`、`str`、`list[str]`、`int|null`。
- 语义意义，例如“输入通道数”“监控指标名”“测试集划分文件”。
- 取值约束，例如 `≥0`、`'max' 或 'min'`、典型范围、类别顺序。
- `null`、`None`、自动检测、延迟初始化、回退逻辑等特殊行为。

### 5. 对齐续行注释

如果某个字段仅靠行尾注释说不清楚，就在下一行或数行继续补充，并保持视觉对齐：

```yaml
gamma: 2.0              # float, Focal Loss 聚焦参数, 典型范围 1.0-3.0
                         # → BinaryFocalLoss.__init__: self.gamma = gamma
```

续行适合承载：

- 经验证的代码消费链。
- 多步影响关系。
- 回退顺序。
- 同一个字段在训练、验证、评估中的多处使用方式。

## `# →` 调用链注释规则

`# →` 是这类配置注释里最有辨识度的标记，但只能在信息已经验证时使用。

### 什么时候应该写 `# →`

- 你已经确认该字段被哪个文件、函数、类或方法读取。
- 你已经确认该字段如何影响实例化、前向传播、损失计算、指标记录、调度器配置或数据集加载。
- 你需要让读者理解“这个值不是孤立配置，它会在运行时影响某段逻辑”。

### `# →` 应该写什么

- 消费入口，例如 `train.py`、`Wrapper.__init__`、`configure_optimizers()`、`validation_step()`。
- 运行时效果，例如存入 `self.hparams`、传入 `instantiate(...)`、作为 `monitor` 参数、决定输出通道数。
- 如果一个字段被多处消费，逐条列出，避免把多个效果糊成一句。

### 什么时候不要写 `# →`

- 你只是猜测它大概率在某处被读到。
- 你只知道“这个字段很重要”，但不知道真正消费逻辑。
- 你没有确认具体符号名，却想强行写出很具体的函数路径。

### `# →` 的约束

- 不要杜撰不存在的代码路径。
- 不要在未验证时乱写代码行号。
- 如果只能确认高层消费者，例如“被 DataModule 读取”，那就保持到这个粒度，不要伪装成精确调用链。

## 应写入的内容类型

YAML 注释应优先覆盖以下信息，而不只是“这个键是什么”。

### 类型与语义

- 字段的数据类型。
- 字段控制的对象或行为。
- 对读者真正有帮助的短语义，例如“前景预测结果”“验证集划分文件”“损失缩放因子”。

### 特殊值与回退逻辑

- `null`、`None`、自动检测、延迟初始化的意义。
- 若存在多级回退顺序，明确写出优先级。
- 如果某个字段只在某种任务模式下有意义，例如二分类、多分类、训练阶段、验证阶段，直接写出来。

### 消费链与运行时行为

- 谁读取它。
- 何时读取。
- 读取后做了什么。
- 是否进一步影响损失、指标、checkpoint 或调度器。

### 列表顺序与映射关系

- 对列表字段，优先写顺序含义、长度约束、位置到语义的映射。
- 对映射字段，写清来源、类别含义、合并策略、背景类位置。

### 示例与替代值

- 对经常需要切换的字段，可以保留注释掉的替代配置作为示例。
- 示例应是“真实可用”的替代值，而不是装饰性的假数据。

### 实验上下文

- 若当前实验有特别意图，例如“本次将所有前景合并为一类”，允许使用 `NOTE:` 或普通块注释直接说明。
- 这类注释可以更贴近当前实验，但不应与通用语义相冲突。

## 常见字段的写法

### `_target_`

写清它实例化什么对象、由谁实例化、实例化时还会被自动传入哪些参数。

### 标量数值字段

优先写类型、含义、单位或典型范围。例如损失参数、调度器步频、类别先验、epsilon。

### 布尔字段

优先写“打开后会发生什么”，而不是只写“是否启用某功能”。

### 路径字段

优先写该路径服务于哪个阶段，是否存在 train/val/test 多阶段分流，以及统一路径的回退逻辑。

### 列表字段

优先写元素类型、顺序约束、长度约束、索引对应关系。若列表承载类别映射、通道排列或 split 文件列表，要显式说明。

### 监控与调度字段

优先写监控指标名、`max/min` 语义、该值会被哪个调度器或 checkpoint 逻辑读取。

## YAML 排版风格

- 文件级横幅和 section 横幅之间保留空行，增强视觉分块。
- 行尾注释与续行注释保持对齐，不要让续行忽左忽右。
- 复杂字段宁可拆成“块注释 + 键值行”，也不要把过长说明硬塞在一行后面。
- 注释掉的替代值保留为合法 YAML 注释，不要写成无法恢复的自然语言段落。
- 保持 YAML 本身仍然清晰可读；注释不应压垮值的视觉层次。

## 避免

- 避免只写“赋值”“参数”“中间配置”“路径”等空泛词。
- 避免把 YAML 键名直接翻译一遍当注释。
- 避免写未经验证的代码路径或虚假的行号。
- 避免把实验性的主观猜测写成确定事实。
- 避免删除用户已经写下的 `NOTE:`、替代值示例或重要上下文。
- 避免对完全自解释的简单键过度注释，导致真正重要的说明被淹没。

## 规范示例

### 示例 1：模型配置

```yaml
# ============================================================
# 某分割模型配置
# 此文件的所有配置项以 cfg.model.* 的形式被 train.py 和 Wrapper 使用
# ============================================================
# _target_: str, 目标类全称; Hydra 据此实例化 Wrapper
#   → train.py: hydra.utils.instantiate(cfg.model, optimizer=..., scheduler=...)
_target_: src.wrappers.volume_segmentation.VolumeSegmentationWrapper

# --- 骨干网络 (Backbone) ---
# backbone: nn.Module, 实例化真正的神经网络
#   → Wrapper.__init__: self.backbone = instantiate(backbone)
backbone:
  _target_: src.model.raunet.SimpleUnet
  in_channels: null      # int|null, 输入通道数; null 表示运行时自动识别
                         # → Wrapper.forward(): 根据第一个 batch 的 x.shape[1] 推断
  out_channels: 1        # int, 输出通道数; 二分类通常为 1
                         # → Wrapper.__init__: 据此决定后续指标对象的类型
  # planes: Sequence[int], 固定长度通道列表
  # 映射关系: (enc0, enc1, enc2, enc3, bottleneck, dec3, dec2, dec1, dec0)
  planes: [64, 256, 256, 256, 256, 256, 128, 64, 64]
```

### 示例 2：数据集配置

```yaml
# ============================================================
# 某数据集配置
# 此文件的所有配置项以 cfg.dataset.* 的形式被 train.py 使用
# ============================================================
# _target_: str, 数据集类全称; Hydra 据此实例化数据集对象
#   → train.py: UnifiedDataModule.setup() 中调用 hydra.utils.instantiate(cfg.dataset, ...)
_target_: src.datasets.dataset.MyDatasets

# --- 数据根目录路径 (允许三个阶段分别指定) ---
# setup() 会优先查找 all_data_path_train / all_data_path_val,
# 若不存在则回退到 all_data_path (统一路径), 再传入数据集构造函数
all_data_path: "/path/to/data"
# all_data_path_train: "/path/to/data"   # str, 训练集数据根目录
# all_data_path_val:   "/path/to/data"   # str, 验证集数据根目录
# all_data_path_test:  "/path/to/data"   # str, 测试集数据根目录

# --- 标签映射 (Label Mapping) ---
# label 来源顺序: [BG, class1, class2, class3]
# NOTE: 当前实验将所有前景类别并为 1 类
class_mapping: [0, 1, 1, 1]

# --- 数据划分文件路径 (Split Files) ---
# split_train / split_val 在 DataModule.setup() 中被显式读取
split_train:
  - "/path/to/train_1.json"
  - "/path/to/train_2.json"
```

## 最终目标

输出的 YAML 注释应当让读者在不跳进代码的情况下，先理解大部分配置语义；当读者需要追到代码时，又能沿着已验证的 `# →` 注释快速定位到真正的消费位置。
