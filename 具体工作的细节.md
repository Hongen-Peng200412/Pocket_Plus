# Pocket_Plus Stage1 实现规格与机制说明

本文档描述 `Pocket_Plus` 中 stage1 训练实现的稳定接口、数据契约、模块职责、信息流与核心机制。

## 文档定位

- 本文档是“实现规格书”，不是“某次实验记录”。
- 本文档优先说明代码中长期稳定存在的接口、张量语义、模块机理、控制边界与前向数据流。
- 配置文件在本文档中的作用，是说明“哪些行为由哪些 config group 控制”，而不是记录某个当前值、默认值、近期实验值或正在训练的组合。
- 若未来实验切换了具体 YAML，但没有改变接口、数据契约、机制语义或模块边界，则本文档通常不需要修改。
- 若代码实现与历史计划、旧 baseline 描述或口头约定冲突，以当前实现为准。

## 规则

- 不将“当前正在训练的配置”写成本文的主叙述。
- 不将“默认值”误写成“机制定义”。
- 当描述一个模块时，优先写清四件事：输入、输出、内部机理、与其它模块的连接关系。
- 当描述一个机制时，优先写清它解决什么问题、状态如何流动、哪些阶段参与、哪些阶段不参与。
- 当描述配置时，只说明“哪个目录控制什么能力”，不强调某个具体配置名此刻取了什么值。

## 1. 训练入口与装配关系

### 1.1 启动链路

训练由 `sbatch` 脚本或直接运行 `src/train.py` 进入。

稳定的职责分层如下：

1. `sbatch/*` 与 `sbatch/_train_core.sh`
   负责集群资源、分布式参数、重试锁文件、最终命令行拼装。
2. `src/train.py`
   负责 Hydra 装配、DataModule、Trainer、logger、checkpoint、batch-size tuning、lazy module materialization。
3. `src/wrappers/voxel_point_stage1.py`
   负责把 backbone、损失函数、优化器、调度器、验证指标与日志整合进 LightningModule。
4. `src/model/stage1_model.py`
   负责 stage1 的真实前向、模块编排、voxel-point 融合、伪原子与 recycle 机制。

### 1.2 `train.py` 的职责

`src/train.py` 不定义领域机制本身，它负责训练过程的运行时装配：

- 解析 Hydra 根配置与 override。
- 构造 `UnifiedDataModule`。
- 通过配置实例化 wrapper。
- 在 DDP 包装前完成 lazy module 的输入通道物化。
- 配置 WandB、checkpoint、progress bar 与 distributed strategy。
- 在启用全局 batch 控制时，先做显存探测，再回填安全的 per-device batch size 与梯度累积步数。

### 1.3 Wrapper 的职责

`src/wrappers/voxel_point_stage1.py` 中的 `VoxelPointStage1Wrapper` 负责：

- 调用 backbone 的 `forward(batch)`。
- 计算 atom 主损失。
- 计算可选的 voxel auxiliary loss。
- 记录训练与验证日志。
- 聚合验证阶段的 atom PR-AUC。
- 根据配置实例化优化器与调度器。

它不重新实现 backbone 的内部机制；它消费 backbone 的输出契约。

## 2. 数据子系统

## 2.1 `BoxPointDataset` 的定位

`src/datasets/box_point_dataset.py` 中的 `BoxPointDataset` 是 stage1 的联合数据集。它的职责是：

- 读取单个 BOX 对应的 voxel 网格、标签和几何元信息。
- 回到对应的 `pdb_id` 读取结构级原子坐标、原子特征与标签。
- 根据当前 BOX 的空间范围，从整条结构中过滤出本 BOX 需要的原子集合。
- 同时构造 voxel 侧监督字段与 atom 侧监督字段。
- 在训练态对 voxel 与 atom 做完全同步的 90 度旋转增强。

换言之，dataset 返回的不是单一 tensor，而是一个完整的 `dict[str, Any]` 样本契约。

## 2.2 单样本返回契约

`BoxPointDataset.__getitem__()` 返回的核心字段分为四类。

### A. Voxel 侧字段

- `voxel_grid`
  形状 `(C, D, H, W)`，`float32`，体素输入特征。
- `voxel_label`
  形状 `(D, H, W)`，`int64`，体素监督标签。
- `hardmask`
  形状 `(D, H, W)`，`int64`，表示哪些 voxel 在几何上对应到真实 core atom。
- `voxel_valid_mask`
  形状 `(D, H, W)`，`bool`，表示哪些 voxel 参与 voxel auxiliary supervision。

### B. BOX 几何字段

- `box_origin_world`
  形状 `(3,)`，`float32`，BOX 左下近角点世界坐标，顺序 `(x, y, z)`。
- `voxel_size_world`
  形状 `(3,)`，`float32`，每个 voxel 的世界尺寸，顺序 `(x, y, z)`。
- `box_shape_zyx`
  形状 `(3,)`，`int64`，体素数组空间尺寸，顺序 `(Z, Y, X)`。

### C. Atom 侧字段

- `atom_coord_world`
  形状 `(N, 3)`，`float32`，绝对世界坐标。
- `atom_coord_local_voxel`
  形状 `(N, 3)`，`float32`，连续 voxel 坐标，顺序仍为 `(x, y, z)`，数值语义是 corner 语义。
- `atom_coord_centered_world`
  形状 `(N, 3)`，`float32`，以 BOX 中心为原点的世界坐标。
- `atom_feat`
  形状 `(N, F_atom)`，`float32`，点分支底层输入特征。
- `atom_label`
  形状 `(N,)`，`int64`，atom 监督标签。
- `atom_is_in_core_box`
  形状 `(N,)`，`bool`，标记该原子是否位于 core BOX 内。
- `atom_valid_mask`
  形状 `(N,)`，`bool`，标记该原子是否参与监督。

### D. 元信息字段

- `sample_name`
- `pdb_id`
- `class_name`
- `instance_id`
- `is_center_box`

这些字段保留 Python 原生类型，供调试、验证与输出使用。

## 2.3 坐标系统

当前实现中同时存在三套原子坐标，它们解决的是不同问题。

### `atom_coord_world`

- 语义：绝对世界坐标。
- 用途：恢复真实空间位置、与结构级数据或伪原子坐标对齐。

### `atom_coord_local_voxel`

- 语义：相对于 BOX 左下近角点的连续 voxel 坐标。
- 顺序：字段顺序保持 `(x, y, z)`，但它描述的是 voxel 网格空间。
- 注意：它是 corner 语义，不是 center-index 语义。
- 用途：与 voxel 网格对齐，尤其用于 voxel->point 采样与 scatter。

### `atom_coord_centered_world`

- 语义：以 BOX 中心为原点的世界坐标。
- 用途：点分支与 atom head 的主坐标表示。

## 2.4 原子筛选与监督语义

### 原子筛选

`BoxPointDataset._select_atoms_for_box()` 将结构级原子划分为：

- core 原子：位于当前 BOX 内部。
- buffer 原子：落在 BOX 外扩 buffer 范围内，但不在 core 内。

这一步决定“哪些原子进入当前样本”，而不是“哪些原子参与监督”。

### `hardmask`

`hardmask` 不是简单的“某类输入通道非零掩码”。当前实现中，它由 core 原子几何投影得到，语义是“哪些 voxel 对应到了真实 core atom 的占位”。

### `voxel_valid_mask`

`voxel_valid_mask` 由 `valid_crop_margin` 控制，在六个面各裁掉一定宽度的边带。它控制 voxel auxiliary loss 的监督范围。

### `atom_valid_mask`

`atom_valid_mask` 同时要求：

- 原子位于 core BOX 内。
- 原子在连续 voxel 坐标意义下落在裁边后的有效区域内。

因此，原子是否进入样本，与它是否参与监督，是两个不同层次的判断。

## 2.5 随机旋转增强

`BoxPointDataset` 在训练态可对以下内容做同步的 90 度旋转：

- `voxel_grid`
- `voxel_label`
- `hardmask`
- `voxel_valid_mask`
- `atom_coord_local_voxel`
- `atom_coord_centered_world`
- `atom_coord_world`

这里的关键要求不是“做了旋转”，而是“voxel 与 atom 在同一几何规则下同步旋转”，否则 voxel->point 融合会系统错位。

## 2.6 `box_point_collate()` 的职责

`src/datasets/box_point_collate.py` 中的 `box_point_collate()` 将单样本字典整理为 batch 字典。

其组织原则是：

- voxel 相关字段在 batch 内形状固定，直接 `stack`。
- atom 相关字段长度可变，沿第 0 维 `cat`。
- 元信息字段保留为 `list`。

collate 额外生成三项索引辅助字段：

- `atom_counts`
  每个样本的 atom 数量。
- `atom_offsets`
  每个样本在展平 atom 序列中的结束偏移。
- `atom_batch_index`
  展平后每个 atom 属于哪个 batch 项。

这三项不是附属细节，而是 point backbone、伪原子注入、mixed/real-only 视图切换和后续按样本还原的基础契约。

## 3. 主模型总装配

## 3.1 `VolumePointStage1Model` 的角色

`src/model/stage1_model.py` 中的 `VolumePointStage1Model` 是 stage1 的总装配 backbone。它负责：

- 编排 embed head、voxel backbone、point backbone 与 atom head。
- 承担 voxel->point 融合。
- 在需要时启用伪原子。
- 在需要时执行 recycle 循环。

它不是单一分支网络，而是整个 stage1 前向系统的调度中心。

## 3.2 输入契约

`VolumePointStage1Model.forward(batch)` 消费的 `batch` 即 `box_point_collate()` 的返回结果。

其中最核心的输入字段是：

- `voxel_grid`
- `voxel_size_world`
- `box_shape_zyx`
- `atom_feat`
- `atom_coord_local_voxel`
- `atom_coord_centered_world`
- `atom_valid_mask`
- `atom_batch_index`
- `atom_offsets`

## 3.3 输出契约

主模型输出中，稳定且重要的字段包括：

- `atom_logits`
  atom 分类 logits。
- `atom_hidden`
  atom head 注意力栈后的隐藏特征。
- `atom_tokens`
  atom head 的输入 token。
- `fused_point_feat`
  最终进入 atom head 的点特征。
- `sampled_point_fusion_feat_dict`
  记录各命名融合点采样得到的 voxel 特征。
- `voxel_logits_aux`
  voxel auxiliary supervision 的 logits。
- `voxel_outputs`
  体素分支输出字典。
- `point_outputs`
  点分支输出字典。
- `embed_output`
  embed head 输出字典；未启用时可为 `None`。
- `recycle_passes_used`
  本次前向实际执行的 recycle 轮数。

## 4. 核心模块机理

## 4.1 `embed head`

文件：`src/model/stage1_embed_head.py`

### 输入

- `atom_feat`
- `atom_coord_centered_world`
- `atom_batch_index`
- `atom_offsets`
- `atom_coord_local_voxel`
- `box_shape_zyx`
- `voxel_size_world`
- `atom_is_in_core_box`

### 输出

- `voxel_pdb_embed_grid`
- `embed_point_feat`
- 裁剪后的 atom 字段
- `global_keep_mask`

### 机理

`embed head` 是一个位于 recycle 循环外的前置模块。它在语义上同时扮演两个角色：

- 原子编码器
- 原子过滤器

它先对输入 atom 做共享 trunk 编码，再分成两条路径：

1. voxel 路径
   将 per-atom hidden 通过 scatter 聚合到 voxel 网格，形成 `voxel_pdb_embed_grid`，供体素分支作为附加输入通道使用。
2. point 路径
   输出 `embed_point_feat`，供点分支输入表示空间做残差增强。

### 渐进裁剪

`embed head` 的关键机制不是单纯“算特征”，而是“逐阶段缩小 buffer 原子集合”。

它在 trunk、voxel path、point path 中都可以按各自的 buffer 半径序列执行裁剪：

- 越靠前的 block，可允许更大的 buffer 范围。
- 越靠后的 block，可逐步收缩到更小的有效原子集合。

因此，`embed head` 不只是把原子信息编码成网格，它还会改变下游 point branch 和监督所面对的 atom 集合。

## 4.2 体素分支

文件：`src/model/stage1_voxel_backbone.py`

### 输入

- `voxel_grid`
- 可选的 `recycle_in`
- 本轮请求返回的命名 feature keys

### 输出

- `voxel_features`
- `voxel_logits_aux`
- `voxel_recycle_out`

### 机理

体素分支基于 RAUNet 风格结构，但其在 stage1 中的角色不只是 voxel 监督头。

它承担三件事：

- 处理体素密度与体素化结构输入。
- 输出多尺度命名体素特征，供点分支在不同层位采样。
- 产生下一轮 recycle 所需的体素隐状态。

因此，体素分支的稳定接口是“命名字典 + aux head + recycle state”，而不是“单一最终 logits”。

## 4.3 点云分支

文件：`src/model/stage1_point_backbone.py`

### 输入

- `atom_feat`
- `atom_coord_centered_world`
- `atom_batch_index`
- `atom_offsets`
- 可选的 `recycle_in`
- 可选的 `point_feature_hook`

### 输出

- `point_feat`
- `point_state`
- `point_recycle_out`
- `point_feature_dict`

### 机理

点分支在进入 PTV3 主干之前，会先把上一轮的 `point_recycle_out` 投影回 atom 特征空间，再与当前 `atom_feat` 融合。这意味着：

- recycle 在点分支中不是外部旁路，而是下一轮输入表示的一部分。
- 点分支读取到的是“当前 atom 特征 + recycle 残差”。

`point_feature_hook` 是点分支的重要扩展接口。主模型利用这个 hook 在命名点层位即时执行 voxel->point 融合，而不是等点分支跑完再一次性融合。

`point_state` 则是 point backbone 专门导出的结构化状态，供 atom head 的 serialized attention 复用。

## 4.4 `atom head`

文件：`src/model/stage1_model.py`

### 输入 token

当前实现中，`atom_tokens` 由下列内容拼接而成：

- 最终点特征 `fused_point_feat`
- `atom_coord_centered_world`
- `atom_valid_mask`

### 内部结构

`atom head` 不只是一个末端 MLP。它的判别过程分为两段：

1. `atom_token_proj`
   先把拼接后的 token 投影到统一隐藏维度。
2. `Stage1SerializedAttentionStack`
   结合点状态做若干层 serialized attention。
3. `atom_logit_head`
   将注意力栈输出映射成最终 logits。

### 机理

因此 atom head 的本质是：

- 在 point branch 已经融合 voxel 信息之后，
- 再以结构化的点序列注意力方式做最终 atom-level 判别。

## 5. 三个核心机制

## 5.1 伪原子机制

文件：`src/model/pseudo_atoms.py`

### 目标

伪原子的作用不是提供监督，而是作为额外消息节点，增强点分支对局部体素与结构上下文的表达能力。

### 生成

`PseudoAtomGenerator.generate()` 会为每个 BOX 现场生成伪原子，主要步骤包括：

- 按每个 BOX 的真实原子数决定目标伪原子数。
- 在 BOX 范围内采样候选位置。
- 可选基于密度图进行加权采样。
- 通过“过近删除 / 过远删除 / 聚类稀疏化”清理候选点。
- 初始化伪原子特征。

### 注入与移除

伪原子不会永久改变 batch 契约，而是通过两步显式切换：

- `inject()`
  按 `[real_i, pseudo_i]` 的交错布局，把伪原子插入 mixed batch。
- `remove()`
  从 mixed batch 恢复 real-only batch。

### 生命周期

伪原子由 `lifecycle = [embed_head, point_backbone, atom_head]` 控制在哪些阶段存在。
当前实现额外要求: 若 `atom_head=True`，则 `point_backbone` 也必须为 `True`；单独只在 atom head 引入伪原子不受支持。

这三个布尔位决定的是阶段边界，不代表某个当前实验是否启用。

### 监督规则

伪原子的 `valid_mask` 为 False，不参与监督损失。
即使伪原子进入 atom head，它们也只参与消息传递，最终输出仍会在需要时裁回真实原子子集。

## 5.2 点云与体素分支的信息融合机制

主要文件：

- `src/model/stage1_model.py`
- `src/model/stage1_point_backbone.py`

### 目标

融合机制的目标是把 voxel 分支中的局部密度上下文注入到点分支的命名层位中，使最终 atom 判别建立在“结构原子表示 + 局部体素上下文”的联合表示上。

### 基本形式

当前主线是单向 `voxel -> point` 融合，不做 `point -> voxel` 回写。

每次融合都遵循同一流程：

1. 根据当前点的 `atom_coord_centered_world` 恢复其 local voxel 连续坐标。
2. 在对应的命名 voxel feature map 上做采样。
3. 将采样得到的 voxel 特征与当前点特征拼接。
4. 通过融合模块投影回点特征维度。

### 挂接方式

融合不是在主模型末端一次性发生，而是通过 `point_feature_hook` 挂接在点分支的命名层位上即时执行。

因此，融合点的定义是“点层位名 -> 体素层位名”的映射关系，而不是硬编码在 point backbone 内部。

### 控制边界

当前实现把融合控制拆成三个维度：

- 哪些点层位要融合
- 每个点层位从哪个 voxel 层位读取
- 每次采样与融合分别采用什么策略

这三部分均由 `configs/model/fusion/*` 控制其能力边界。

## 5.3 recycle 机制

主要文件：

- `src/model/stage1_model.py`
- `src/model/stage1_voxel_backbone.py`
- `src/model/stage1_point_backbone.py`

### 目标

recycle 机制允许模型在同一个前向中进行多轮“体素分支 -> 点分支 -> atom 判别”的迭代细化。

### 轮数语义

recycle 轮数的决定逻辑是：

- 若关闭 recycle，则固定执行 1 轮。
- 若启用 recycle 且处于训练态，可按配置决定是否对轮数做随机采样。
- 否则执行配置允许的完整轮数。

### 状态流

体素分支跨轮传递的是：

- `voxel_recycle_out`

点分支跨轮传递的是：

- real-only 视图下的 `point_recycle_out`

若伪原子启用了相应 recycle 策略，还可能存在独立保存的伪原子 point recycle 状态。

### 重要边界

- `embed head` 不参与 recycle；它始终在循环外只执行一次。
- `detach_recycle_states` 决定跨轮状态是否截断梯度。
- 伪原子是否跨轮保留位置、特征或 point recycle 隐状态，不由 recycle 本身决定，而由 `pseudo atom` 的 `recycle_policy` 决定。

## 6. 训练与监督

## 6.1 监督结构

当前 wrapper 支持两类监督：

- atom 主监督
- voxel auxiliary supervision

atom 主监督直接作用于 `atom_logits`。
voxel auxiliary supervision 直接作用于 `voxel_logits_aux`。

## 6.2 损失与日志的职责边界

损失函数的具体实现由 `configs/loss/*` 控制。
wrapper 只负责：

- 读取 backbone 输出与 batch 真值。
- 调用损失函数。
- 记录 `train/*` 与 `val/*` 指标。

因此，“损失函数取什么超参”属于配置控制面；“atom 主损失与 voxel 辅助损失这两条监督路径存在并被加权汇总”属于实现规格。

## 6.3 验证指标

验证阶段的核心结构化指标是 atom 级别的 PR-AUC。它反映的是：

- 在 atom supervision 的有效掩码下，
- 模型对 pocket atom 的排序与识别能力。

具体监控哪一个日志键，由 wrapper 与 model config 的控制面共同决定，但“验证主要围绕 atom-level 预测质量展开”是当前实现的稳定设计。

## 7. 配置控制面

本节只说明“哪个配置目录控制什么”，不说明某个当前值。

### `configs/model/voxel_backbone/*`

控制体素分支的：

- 输入通道物化方式
- 主干结构
- 可导出的命名体素层位
- voxel auxiliary head 相关结构
- voxel recycle 输出结构

### `configs/model/point_backbone/*`

控制点分支的：

- 输入 atom 特征维度
- 点主干结构
- embedding / CPE / PTV3 参数
- recycle 输入注入方式
- 输出点特征维度

### `configs/model/fusion/*`

控制融合机制的：

- 点层位与体素层位的映射关系
- 采样策略
- 融合策略
- 融合层的投影结构

### `configs/model/embed_head/*`

控制 embed head 的：

- 共享 trunk 结构
- voxel path / point path 输出通道
- 渐进裁剪半径序列
- scatter 聚合方式
- 点路径是否输出额外表示

### `configs/model/pseudo_atom/*`

控制伪原子机制的：

- 每个 BOX 的伪原子规模
- 位置采样与删除规则
- 特征初始化方式
- 生命周期
- recycle policy

### `configs/model/atom_head/*`

控制 atom head 的：

- token 隐藏维度
- serialized attention 结构
- CPE/FFN/regularization 相关参数
- 最终分类头的容量

### `configs/dataset/*`

控制数据子系统的：

- 读取哪些数据目录
- 标签映射
- 原子 buffer 半径
- valid crop 规则
- 数据增强开关

### `configs/train/*`

控制训练过程的：

- DataLoader 相关参数
- distributed 相关参数
- 全局 batch 与自动 tuning 策略
- optimizer / scheduler
- fault tolerance

### `configs/loss/*`

控制监督系统的：

- atom loss 类型与参数
- voxel auxiliary loss 类型与参数
- 各项损失的权重

## 8. 维护规则

未来维护本文档时，遵循以下规则：

- 若只更换具体实验配置，但没有改变接口、机制或数据契约，本文通常无需更新。
- 若新增模块，应优先补“输入、输出、状态流、控制边界”，而不是先补某个实验值。
- 若修改了张量 shape、字段语义、mixed/real-only 视图规则、伪原子生命周期、融合路径或 recycle 状态流，则必须同步更新本文。

- 不得把“某个 YAML 当前值”写成“代码机制的定义”。

## 9. 一句话总结

当前 stage1 不是单一 voxel 分割网络，而是一个以联合样本契约为基础、通过 embed head、体素分支、点分支、atom head、伪原子与 recycle 共同完成 atom-level 判别的装配式系统；本文档记录的是这个系统如何工作，而不是某次实验恰好怎样配置。
