# 1. 训练入口与配置组合

## 1.1 真实启动链路

当前训练通过 `sbatch/<gpu_profile>/*.sbatch` 或直接调用 `src/train.py` 进入。

真实启动链路是：

1. 进入某个 `sbatch` 脚本。
2. 脚本加载 `sbatch/_common.sh` 与 `sbatch/_train_core.sh`。
3. `_train_core.sh` 调用：
   `python Pocket_Plus/src/train.py --config=<CONFIG_NAME> <HYDRA_OVERRIDES> train.nnodes=<NNODES> train.devices=<DEVICES>`
4. `src/train.py` 用 `@hydra.main(config_path="../configs", config_name=_CONFIG_NAME)` 启动 Hydra。
5. Hydra 从 `configs/<CONFIG_NAME>.yaml` 解析根配置，并叠加命令行 override。

`_train_core.sh` 的参数解析规则固定为：

- 第一个参数若以 `+` 开头，则它被视为 Hydra override，此时 `CONFIG_NAME` 自动取 `base`。
- 第一个参数若不是 `+...`，则它就是 `--config` 对应的根配置名。
- `train.nnodes` 与 `train.devices` 会被 sbatch 脚本最终覆盖。

因此，当前完整且可直接使用的命令形态是：

- `python Pocket_Plus/src/train.py --config=base +experiment=exp001_baseline`
- `sbatch sbatch/h100/1gpu.sbatch "+experiment=exp001_baseline"`
- `sbatch sbatch/h200/1gpu.sbatch "+experiment=exp001_baseline"`



## 1.2 根配置与具名实验

`configs/base.yaml` 是 Hydra 根骨架，负责选择各个 config group：

1. `model: default`
2. `model/voxel_backbone: raunet_64`
3. `model/point_backbone: ptv3_dense`
4. `model/fusion: dense_4hook`
5. `model/atom_head: 4layer`
6. `model/pseudo_atom: none`
7. `model/embed_head: none`
8. `loss: focal_default`
9. `dataset: fused`
10. `post_process: none`
11. `output: standard`
12. `train: default_h100`

`configs/experiment/exp001_baseline.yaml` 是当前仓库内的完整具名实验，它补齐 `VolumePointStage1Model` 所需的总装配级参数，并把训练 profile 覆盖为 `default_h200`。

该实验当前定义的是：

- 体素分支：`raunet_64`
- 点云分支：`ptv3_dense`
- 融合：`dense_4hook`
- `atom head`：`4layer`
- `pseudo atom`：关闭
- `embed head`：关闭
- 数据集：`fused`
- 损失：`focal_default`
- 训练 profile：`default_h200`


## 1.3 Batch 语义

本文档中的 batch 术语固定采用下面的定义：

- `batch_size` 指 `cfg.train.batch_size`，即每张卡的 batch size。
- `world size` 指 `train.devices * train.nnodes`。
- `effective batch size` 指 `batch_size * accumulate_grad_batches * world size`。

当前两个训练 profile 为：

- `default_h100`: `batch_size=3`, `accumulate_grad_batches=12`
- `default_h200`: 基于 `default_h100` 覆盖为 `batch_size=6`, `accumulate_grad_batches=6`


# 数据子系统

## 1. 数据集定位

当前主数据集是 `src/datasets/box_point_dataset.py` 中的 `BoxPointDataset`。

它负责一次性返回 stage1 训练所需的完整样本字典，包含：

- voxel 输入
- voxel 标签
- voxel 辅助监督掩码
- atom 坐标
- atom 特征
- atom 标签
- atom 监督掩码
- 样本级元信息



## 2. 当前 `fused` 配置

`configs/dataset/fused.yaml` 当前主配置为：

- `_target_: src.datasets.box_point_dataset.BoxPointDataset`
- `data_folder_names = ["emdb_BOX", "pdb_feature_BOX", "pdb_label_BOX"]`
- `class_folder_names = ["metal_ion", "peptide", "nucleic", "small_molecule"]`
- `class_mapping = [0, 1, 0, 0, 1]`
- `atom_buffer_radius = 0.0`
- `valid_crop_margin = 0`
- `enable_random_rotation = true`

`class_mapping = [0, 1, 0, 0, 1]` 的当前含义是：

- 原始标签 `0` 映射到 `0`
- 原始标签 `1` 映射到 `1`
- 原始标签 `2` 映射到 `0`
- 原始标签 `3` 映射到 `0`
- 原始标签 `4` 映射到 `1`

因此当前训练虽然读取的是多类 `pocket_class_ids`，但经过映射后是二分类训练。


## 3. 样本名解析

当前样本名格式固定为：

`{pdb_id}_{instance_id}_{rxx}_{ryy}_{rzz}[_C]`

`BoxPointDataset._parse_sample_name()` 当前会解析出：

- `pdb_id`
- `instance_id`
- `rxx`
- `ryy`
- `rzz`
- `is_center_box`

其中 `pdb_id` 会统一转为小写。


## 4. BOX 侧数据读取

`_load_box_npz_triplet()` 的当前行为固定为：

1. 遍历 `data_folder_names`
2. `label` 文件夹读成 `voxel_label`
3. 其它文件夹沿 channel 维拼接成 `voxel_grid`
4. 含 `emdb` 的通道会做整图 z-score
5. 从 npz 中读取或反推：
   - `box_origin_world`
   - `voxel_size_world`
   - `x_range`
   - `y_range`
   - `z_range`
6. 基于非 EMDB 通道是否非零，生成 `hardmask`

当前 `hardmask` 的语义是：

- 只看非 EMDB feature 通道
- 某体素只要有任一非 EMDB feature 非零，则 `hardmask=1`
- 若没有任何非 EMDB feature 通道，则返回全零 `hardmask`

注意：

- `x_range / y_range / z_range` 当前会在内部读取，但不会进入最终 batch 主契约。
- 文档中不要把它们写成 `__getitem__()` 的返回字段。


## 5. 原子侧数据读取与缓存

`_load_structure_npz_cached()` 当前从：

- `atoms.npz`
- `labels.npz`

中读取：

- `coords -> atom_coord_world`
- `features -> atom_feature_raw`
- `binding_mask`
- `pocket_class_ids`
- `instance_ids`

但当前真正进入主训练标签的是：

- `pocket_class_ids`

不是：

- `binding_mask`

`binding_mask` 当前只是随原始标签一起被读取，并未作为主损失标签使用。

当前实现还维护每个 DataLoader worker 内部的轻量 LRU cache：

- key 为 `pdb_id`
- value 为对应结构的原子与标签字典


## 6. 原子筛选规则

`_select_atoms_for_box()` 当前固定按几何规则筛选：

1. 先用当前 BOX 的 `[origin, max)` 定义 `core_mask`
2. 再用 `atom_buffer_radius` 做外扩，得到 `selected_mask`
3. 返回：
   - `selected_idx`
   - `atom_is_in_core_box`

在当前主配置下：

- `atom_buffer_radius = 0.0`

因此现行训练里：

- `selected_mask == core_mask`
- 不存在额外 buffer atom

也就是说，代码支持 buffer 原子；在当前默认数据配置下，这部分能力没有被实际用到。


## 7. 原子标签规则

当前原子标签来自：

1. `labels.npz["pocket_class_ids"]`
2. 再经过 `_apply_class_mapping()`

因此当前 `atom_label` 是：

- `selected_idx` 对应原子的类别 ID
- 经过 `class_mapping` 后的最终训练标签

当前主线不要再写成：

- “原子主标签直接用 `binding_mask`”


## 8. 原子特征规则

当前 `atom_feat` 由 `_build_atom_features()` 生成，第一版保持为：

- 从 `atom_feature_raw[selected_idx]` 直接取出
- dtype 为 `float32`
- 当前主配置下特征维度是 `49`

以下内容当前都没有被拼进底层 `atom_feat`：

- `atom_is_in_core_box`
- `atom_valid_mask`
- 相对 BOX 中心的几何坐标

这些信息的当前处理方式分别是：

- `atom_is_in_core_box` 作为单独字段保留
- `atom_valid_mask` 作为单独字段保留，并在 atom token 阶段再拼接
- 几何坐标通过 `atom_coord_centered_world` 单独提供


## 9. 坐标系统

当前代码中需要严格区分三套坐标：

### 9.1 `atom_coord_world`

- 形状 `(N_selected, 3)`
- 顺序 `(x, y, z)`
- 世界坐标系

### 9.2 `atom_coord_local_voxel`

- 形状 `(N_selected, 3)`
- 顺序仍是 `(x, y, z)`
- 连续 voxel 坐标
- 语义是 corner 语义，不是 center-index 语义

定义为：

`(atom_coord_world - box_origin_world) / voxel_size_world`

若某原子恰好落在体素中心，则该坐标会带 `0.5` 偏移。

### 9.3 `atom_coord_centered_world`

- 形状 `(N_selected, 3)`
- 顺序 `(x, y, z)`
- 以 BOX 中心为原点的世界坐标

当前 point 分支与 voxel->point 采样主链路，使用的是：

- `atom_coord_centered_world`

当前 `stage1_model.py` 在采样前会从：

- `atom_coord_centered_world`
- `voxel_size_world`
- `box_shape_zyx`

重新恢复出局部 voxel 坐标，再送入 `grid_sample()`。

因此：

- `atom_coord_local_voxel` 目前属于数据契约、调试字段和未来扩展接口
- 当前主线采样时并没有直接读取 batch 里的 `atom_coord_local_voxel`


## 10. 监督掩码

### 10.1 `voxel_valid_mask`

由 `_build_voxel_valid_mask()` 生成。

规则是：

- 初始为全 `True`
- 在六个面各裁掉 `valid_crop_margin` 个 voxel

当前主配置：

- `valid_crop_margin = 0`

因此当前主线中：

- `voxel_valid_mask` 全为 `True`

### 10.2 `atom_valid_mask`

由 `_build_atom_valid_mask()` 生成。

当前规则是：

1. 原子必须在 `core box` 内
2. 原子连续 voxel 坐标必须落在裁边后的有效区域内

在当前主配置下：

- `atom_buffer_radius = 0.0`
- `valid_crop_margin = 0`

因此当前主线中：

- `atom_valid_mask == atom_is_in_core_box`
- 又由于当前没有 buffer atom，实际等价于“当前样本中所有被选中原子都参与监督”

必须注意：

- 这是“当前配置结果”
- 这只是当前配置的结果；代码本身仍支持 buffer 与 crop


## 11. 随机旋转增强

当前训练态支持对 voxel 与 atom 做完全同步的 90 度旋转增强。

同步旋转覆盖的字段包括：

- `voxel_grid`
- `voxel_label`
- `hardmask`
- `voxel_valid_mask`
- `atom_coord_local_voxel`
- `atom_coord_centered_world`
- `atom_coord_world`
- `box_shape_zyx`
- `atom_valid_mask`

当前实现还带有两个约束：

1. `voxel_size_world` 必须近似各向同性
2. 若是奇数次 90 度轴交换，旋转平面的两轴长度必须一致

当前主配置里：

- `enable_random_rotation = true`

仅在训练模式下启用。


## 12. `__getitem__()` 当前返回契约

当前样本字典返回：

- `voxel_grid`: `(C, D, H, W)`, `float32`
- `voxel_label`: `(D, H, W)`, `int64`
- `hardmask`: `(D, H, W)`, `int64`
- `voxel_valid_mask`: `(D, H, W)`, `bool`
- `box_origin_world`: `(3,)`, `float32`
- `voxel_size_world`: `(3,)`, `float32`
- `box_shape_zyx`: `(3,)`, `int64`
- `atom_coord_world`: `(N, 3)`, `float32`
- `atom_coord_local_voxel`: `(N, 3)`, `float32`
- `atom_coord_centered_world`: `(N, 3)`, `float32`
- `atom_feat`: `(N, 49)`, `float32`
- `atom_label`: `(N,)`, `int64`
- `atom_is_in_core_box`: `(N,)`, `bool`
- `atom_valid_mask`: `(N,)`, `bool`
- `sample_name`
- `pdb_id`
- `class_name`
- `instance_id`
- `is_center_box`

# 3. 模型子系统

模型总装配由 `src/model/stage1_model.py` 中的 `VolumePointStage1Model` 负责。下文按模块顺序描述当前行为：

1. `embed head`
2. 体素分支
3. 点云分支
4. 体素与点云分支的信息融合机制
5. 伪原子
6. `atom head`
7. recycle 机制

## 3.1 `embed head`

文件与符号：

- `src/model/stage1_embed_head.py`
- `trim_buffer_atoms()`
- `scatter_to_voxel_grid()`
- `Stage1EmbedHead`

`Stage1EmbedHead` 是一个前置模块。它接收 atom 级输入，在点上执行共享编码，再分叉产生：

- 体素嵌入网格 `voxel_pdb_embed_grid`
- 可选的点特征 `embed_point_feat`
- 裁剪后的 atom 字段
- 全局保留掩码 `global_keep_mask`

输入契约：

- `atom_feat`: `(sumN, F_atom)`
- `atom_coord_centered_world`: `(sumN, 3)`
- `atom_batch_index`: `(sumN,)`
- `atom_offsets`: `(B,)`
- `atom_coord_local_voxel`: `(sumN, 3)`
- `box_shape_zyx`: `(B, 3)`
- `voxel_size_world`: `(B, 3)`
- `atom_is_in_core_box`: `(sumN,)`

初始化结构固定包含：

1. `input_proj`
2. `trunk_blocks`
3. `voxel_blocks`
4. `voxel_out_proj`
5. 可选 `point_blocks`
6. 可选 `point_out_proj`

前向行为固定为：

1. 先将原子特征投影到 `embed_hidden_dim`。
2. 构造 `Point` 对象，并按 `serialization_orders` 序列化。
3. 进入共享 trunk，每个 block 结束后调用 `trim_buffer_atoms()`。
4. `trim_buffer_atoms()` 通过 `atom_coord_local_voxel * voxel_size_world` 恢复局部世界坐标，并计算点到 core box 的 `L∞` 距离；超出该层 `buffer radius` 的 buffer 原子会被裁掉。
5. 体素分支复制 trunk 结果，执行 `voxel_blocks`，每层后继续裁剪。
6. 体素分支把 per-atom hidden 投影成 `embed_voxel_out_channels`，再通过 `scatter_to_voxel_grid()` 聚合到 `(B, C, D, H, W)`。
7. 若 `embed_point_out_channels > 0`，点分支也从 trunk 结果分叉，执行 `point_blocks`，并输出 `embed_point_feat`。
8. 若存在点分支，则点分支的裁剪结果决定最终返回给主模型的 atom 字段；否则使用 trunk 的裁剪结果。

输出契约：

- `voxel_pdb_embed_grid`: `(B, embed_voxel_out_channels, D, H, W)`
- `embed_point_feat`: `(sumN', embed_point_out_channels)` 或 `None`
- `atom_feat`: `(sumN', F_atom)`，这里保留的是裁剪后的原始 atom 特征
- `atom_coord_centered_world`: `(sumN', 3)`
- `atom_batch_index`: `(sumN',)`
- `atom_offsets`: `(B,)`
- `atom_coord_local_voxel`: `(sumN', 3)`
- `atom_is_in_core_box`: `(sumN',)`
- `global_keep_mask`: `(sumN,)`

主模型在集成 `embed head` 且 `embed_head.has_point_output=True` 时，会额外建立：

- `embed_point_add_proj`
- `embed_point_gate`

随后执行：

`batch["atom_feat"] = embed_point_add_proj(atom_feat) + embed_point_gate * embed_point_feat`


## 3.2 `体素分支`

文件与配置：

- `src/model/stage1_voxel_backbone.py`
- `configs/model/voxel_backbone/raunet_64.yaml`

体素分支负责把 `batch["voxel_grid"]` 编码成多尺度稠密 3D 特征图，并额外输出 voxel 级辅助预测与下一轮 recycle 使用的体素状态。该模块通过 `VolumePointStage1Model.set_input_channels()` 做 lazy 初始化，因此真实输入通道数由两部分组成：

- 数据集提供的原始 `voxel_grid` 通道数
- 若启用 `embed head`，再额外加上 `embed_voxel_out_channels`

默认实现是 `Stage1VoxelBackbone(SimpleUnet)`。`raunet_64.yaml` 中的关键结构参数为：

- `feature_channels: 64`
- `planes: [64, 256, 256, 256, 256, 256, 128, 64, 64]`
- `gradient_checkpoint: true`
- `return_feature_keys: ["voxel_ds_2", "voxel_ds_3", "voxel_c4", "voxel_final"]`
- `aux_head_hidden_channels: 32`

前向行为可以概括为：

1. 读取 `voxel_grid` 作为主输入；若存在 `embed_output["voxel_pdb_embed_grid"]`，则在进入 backbone 前沿通道维拼接。
2. 读取上一轮 recycle 传回的 `voxel_recycle_in`；若当前没有可复用状态，则内部退化为零输入。
3. 经过编码器 / 解码器主链路，得到一组层级清晰的稠密体素特征。
4. 将被请求的层位统一整理到 `voxel_features` 字典中，供后续 voxel-to-point 融合按名称查找。
5. 以 `voxel_final` 为输入，同时生成：
   - `voxel_logits_aux`：体素级辅助监督头
   - `voxel_recycle_out`：下一轮 recycle 使用的体素状态

代码内部会维护更完整的一组命名特征，例如：

- `voxel_ds_0`
- `voxel_ds_1`
- `voxel_ds_2`
- `voxel_ds_3`
- `voxel_ds_4`
- `voxel_c4`
- `voxel_c3`
- `voxel_c2`
- `voxel_c1`
- `voxel_c0`
- `voxel_final`

但主模型只会显式请求两类特征来源的并集：

- `voxel_backbone.return_feature_keys`
- `point_fusion_map` 中被配置引用到的 voxel 层位

因此，体素分支的职责是稳定地产生可采样的稠密 3D 表示；它本身不做逐点融合，只负责把后续需要的体素特征准备好。


## 3.3 `点云分支`

文件与配置：

- `src/model/stage1_point_backbone.py`
- `configs/model/point_backbone/ptv3_dense.yaml`

点云分支负责把 atom 级输入编码成逐原子的点特征，并在内部多个层位上显式暴露命名 hook，供融合模块与 atom head 使用。默认 backend 为 `ptv3`，另提供一个接口兼容用的 `zeros` backend。

默认配置中的关键参数包括：

- `backend: ptv3`
- `atom_feature_dim: 49`
- `point_grid_size: 0.25`
- `input_embed_dim: 64`
- `input_embed_hidden_dim: 128`
- `out_channels: 64`
- `recycle_feature_dim: 64`
- `serialization_orders: ["z", "z-trans", "hilbert", "hilbert-trans"]`
- `embedding_impl: pointconv`
- `cpe_impl: pointconv`
- `enc_depths: [2, 2, 2, 6, 2]`
- `dec_depths: [2, 2, 2, 2, 0]`

输入契约固定为：

- `atom_feat`
- `atom_coord_centered_world`
- `atom_batch_index`
- `atom_offsets`
- `point_recycle_in`

点云分支当前的 recycle 融合已经不是“直接拼接再投影”，而是更接近 AF2 的残差式写法。进入主干之前，代码会按如下顺序处理输入：

1. 若 `point_recycle_in` 为空，则先退化为与当前 atom 数量对齐的零张量。
2. 按配置决定是否对 `point_recycle_in` 做归一化；当前支持：
   - `recycle_in_norm_mode: none`
   - `recycle_in_norm_mode: layernorm`
3. 用 `recycle_input_proj` 将 recycle 状态投影回当前 `atom_feat` 的特征空间。
4. 若 `recycle_use_gate=true`，则再乘上逐通道可学习门控 `recycle_input_gate`。
5. 将得到的 recycle 残差与当前 `atom_feat` 直接相加。
6. 把残差融合后的 atom 特征送入 `atom_input_proj`，得到真正进入 point backbone 的 `point_input_feat`。

这一路径的功能描述是：上一轮 point 分支产出的表征，不再被当成“附加输入通道”硬拼接，而是作为一种可控的残差修正项，帮助当前轮在原始 atom 特征之上做迭代细化。这样做可以尽量保持主干接口稳定，同时让 recycle 信息的注入强度更容易控制。

随后模型构造 `Point` 结构，统一携带：

- 点特征
- 世界坐标
- `batch` / `offset`
- `grid_size`

默认 `ptv3` 路径的计算顺序为：

1. 在 `point_input_feat` 位置导出输入投影后的特征。
2. 调用 `serialization()`，按配置顺序生成序列化表示。
3. 调用 `sparsify()`，构造 PTV3 所需的稀疏结构。
4. 经过输入 embedding，导出 `point_embed`。
5. 依次穿过 encoder stages，并在各 stage 结束后触发 `point_enc*` hook。
6. 依次穿过 decoder stages，并在各 stage 结束后触发 `point_dec*` hook。
7. 导出最终的 `point_feat` 作为点云分支主输出。

当前稳定的命名点变量包括：

- `point_input_feat`
- `point_embed`
- `point_enc1`
- `point_enc2`
- `point_enc3`
- `point_enc4`
- `point_enc5`
- `point_dec1`
- `point_dec2`
- `point_dec3`
- `point_dec4`
- `point_feat`

除最终逐点特征外，点云分支还返回：

- `point_state`：保留序列化后的点状态，供 atom head 的序列化注意力栈复用
- `point_feature_dict`：收集被请求导出的中间点特征
- `point_recycle_out`：最终逐点特征，可在下一轮 recycle 中再次输入

边界情况下，点云分支也保持明确接口：

- 若 batch 中没有 atom，直接构造形状正确的零输出
- 若 backend 配成 `zeros`，则跳过 PTV3 计算，返回零特征，但仍保留合法的 `point_state`


## 3.4 `体素与点云分支的信息融合机制`

文件与配置：

- `src/model/stage1_model.py`
- `configs/model/fusion/dense_4hook.yaml`

体素与点云之间的信息融合由 `VolumePointStage1Model` 统一调度。它不在 voxel backbone 内完成，也不写死在 point backbone 内部，而是通过“点分支命名 hook + 体素采样 + 学习式投影”这一套机制完成逐层融合。

当前融合配置由三组映射共同定义：

- `point_fusion_map`：指定哪个点层位读取哪个 voxel 层位
- `sampler_modes`：指定该层位使用 `trilinear` 还是 `nearest`
- `point_fusion_modes`：指定采样后如何与当前点特征融合

默认 `dense_4hook` 配置如下：

- `point_enc2 <- voxel_ds_2`
- `point_enc3 <- voxel_ds_3`
- `point_enc4 <- voxel_c4`
- `point_feat <- voxel_final`

四个 hook 位的默认融合策略一致：

- `sampler_mode: trilinear`
- `point_fusion_mode: concat_linear`

其核心流程为：

1. 点云分支在 `point_input_feat / point_embed / point_enc* / point_dec* / point_feat` 等命名层位运行时，统一通过 hook 回调进入 `VolumePointStage1Model._fuse_point_variable()`。
2. 若当前层位在 `point_fusion_map` 中有映射，模型就取出对应的稠密体素特征图。
3. 模型并不直接消费 batch 中现成的 `atom_coord_local_voxel`，而是先用：
   - `atom_coord_centered_world`
   - `voxel_size_world`
   - `box_shape_zyx`
   恢复当前点在局部体素坐标系中的连续坐标。
4. 连续 voxel 坐标会进一步转换成 `grid_sample()` 所需的归一化坐标；当前 `trilinear` 模式在 5D `grid_sample` 中对应 `mode="bilinear"`。
5. 对每个 box 单独做体素特征采样，再把采样结果按 atom 顺序拼回整体 batch。
6. 将“当前点特征”和“采样到的体素特征”在特征维拼接，并送入对应层位的融合投影模块。
7. 融合结果回写到当前点变量，继续流向后续 point backbone 或 atom head。

当前已落地的融合模块只有一种：

- `concat_linear`

它对应一组按层位独立维护的 `point_fusion_modules`。每个模块都接收：

- 当前点层位特征
- 与之对齐采样得到的 voxel 特征

再输出与该点层位通道数相匹配的融合结果。融合过程中，模型还会把采样得到的逐点 voxel 特征额外保存在：

- `sampled_point_fusion_feat_dict`

便于调试和分析。

需要单独注意的是，`embed head` 产生的 `embed_point_feat` 不属于这里的 voxel-to-point hook 融合。它是 recycle 循环开始前的一次性点特征残差注入；而本节描述的是体素主干与点云主干之间的逐层在线融合。


## 3.5 `伪原子`

文件与配置：

- `src/model/pseudo_atoms.py`
- `configs/model/pseudo_atom/none.yaml`
- `configs/model/pseudo_atom/32_2.yaml`
- `configs/model/pseudo_atom/64_3.yaml`

伪原子由 `PseudoAtomGenerator` 提供，是主模型中的可选增强模块。启用后，模型会在真实 atom 之外，为每个 box 生成一批附加的 pseudo atom，用于扩展点集覆盖范围，并按配置决定它们参与哪些阶段。

当前 `uniform_32` 配置的关键参数包括：

- `base_count: 32`
- `scale_factor: 0.5`
- `max_sample_rounds: 3`
- `init_feat_mode: zero`
- `neighbor_radius: 3.0`
- `enable_density_weighting: false`
- `lifecycle: [false, true, true]`

生成逻辑主要发生在 CPU / numpy 侧，基本步骤为：

1. 以每个 box 内的真实原子数为基准，计算目标伪原子数量：
   - `target_count = base_count + scale_factor * N_real`
2. 在当前 box 空间内采样候选位置；若开启 `enable_density_weighting`，则可参考 `voxel_grid` 指定通道做密度加权。
3. 根据删除规则裁掉不合适的候选点，例如：
   - `delete_too_close_radius`
   - `delete_too_far_radius`
4. 按 `init_feat_mode` 初始化伪原子特征，当前支持：
   - `zero`
   - `neighbor_mean`
5. 为伪原子分配固定的元信息：
   - `pseudo_label = 0`
   - `pseudo_valid_mask = false`

伪原子并不是简单地一次拼进去后永久保留。主模型会使用 `inject()`、`remove()` 和 `build_real_mask()` 在不同阶段维护真实原子与伪原子的对应关系：

- `inject()`：按 box 粒度把伪原子插入到真实原子序列中
- `remove()`：按保存的 `split_info` 恢复真实原子子集
- `build_real_mask()`：在需要保留真实原子输出时构造布尔掩码

`lifecycle` 决定伪原子是否进入点云分支、是否保留到 atom head、以及在阶段切换时是否及时回收。因此，伪原子对模型的影响不是单一的“加点”，而是一个受生命周期调度控制的阶段性扩展机制。


## 3.6 `atom head`

文件与配置：

- `src/model/stage1_model.py`
- `configs/model/atom_head/4layer.yaml`

`atom head` 负责把已经融合完成的逐点特征转换成最终的 atom 级 logits。它位于 voxel backbone、point backbone 和 voxel-to-point 融合之后，是整个 stage1 主预测头。

默认 `4layer` 配置的关键参数为：

- `atom_head_hidden_dim: 128`
- `atom_head_num_heads: 4`
- `atom_head_patch_size: 128`
- `atom_head_num_layers: 4`
- `atom_head_attn_drop: 0.2`
- `atom_head_proj_drop: 0.2`
- `atom_head_cpe_impl: none`
- `atom_head_drop_path: 0.0`
- `atom_head_pre_norm: true`

进入 `atom head` 之前，主模型先构造 atom token。当前 token 的定义是：

1. `fused_point_feat`
2. `atom_coord_centered_world`
3. `atom_valid_mask.float().unsqueeze(-1)`

即：

`atom_tokens = concat(fused_point_feat, atom_coord_centered_world, atom_valid_mask)`

这里拼接的是 `atom_valid_mask`，不是 `atom_is_in_core_box`。随后执行：

1. 用 `atom_token_proj` 将 token 投影到 `atom_head_hidden_dim`
2. 把投影后的 hidden 与 `point_state` 一起送入 `Stage1SerializedAttentionStack`
3. 通过多层序列化注意力块更新逐 atom hidden
4. 用 `atom_logit_head` 输出最终 `atom_logits`

`Stage1SerializedAttentionStack` 当前以 PTV3 `Block` 为每层基本单元。每层都围绕同一份 `point_state` 工作，因此 atom head 复用了点云分支已经建立好的序列化结构，而不是重新做一套点集组织。最终前向输出中与该部分直接相关的键包括：

- `atom_tokens`
- `atom_hidden`
- `atom_logits`

若某一轮前向中引入了伪原子，atom head 可能先在“真实原子 + 伪原子”的联合点集上计算，再在输出阶段通过真实原子掩码恢复回真实监督子集。


## 3.7 `recycle 机制`

文件与配置：

- `src/model/stage1_model.py`
- `configs/experiment/exp001_baseline.yaml`

recycle 机制负责在同一前向内重复运行“体素分支 -> 点云分支 -> atom head”主链路，并把上一轮得到的中间状态作为下一轮输入的一部分。当前配置由以下参数控制：

- `enable_recycling: true`
- `max_recycles: 3`
- `randomize_recycles: true`
- `detach_recycle_states: true`

轮数选择规则固定为：

- 若 `enable_recycling=false`，则只运行 1 轮
- 若处于训练态且 `randomize_recycles=true`，则从 `1..max_recycles` 中均匀采样本次前向轮数
- 否则固定运行 `max_recycles` 轮

每轮结束后，模型都会产生两份可复用状态：

- `voxel_recycle_out`
- `point_recycle_out`

下一轮会分别把它们作为：

- 体素分支的 `voxel_recycle_in`
- 点云分支的 `point_recycle_in`

若 `detach_recycle_states=true`，则这两份状态会在跨轮传递前做 `detach()`，从而阻断梯度跨 recycle 轮回流。这样每轮都能利用上一轮的表征结果，但不会把整条多轮链路无约束地连成更长的反向图。

还有两个实现细节需要明确：

1. `embed head` 若启用，只在 recycle 循环外执行一次，不会每轮重复生成。
2. 伪原子若启用，其注入、移除与恢复会跟随 recycle 循环逐轮执行，并保持与真实原子索引的一致关系。

模型最终会把本次实际使用的轮数记录到：

- `recycle_passes_used`


## 3.8 当前前向主链路

当前真实前向链路固定为：

1. `BoxPointDataset.__getitem__()` 返回 dict sample
2. `box_point_collate()` 组成 dict batch
3. （可选）`Stage1EmbedHead` 在 recycle 循环外执行一次：
   - 裁剪 buffer 原子
   - scatter 生成 `voxel_pdb_embed_grid`
   - （可选）残差加和到 `atom_feat`
   - 更新 batch 内 per-atom 字段
4. `Stage1VoxelBackbone` 输出：
   - `voxel_features`（输入通道可能因 embed head 而扩展）
   - `voxel_logits_aux`
   - `voxel_recycle_out`
5. `Stage1PointBackbone` 在命名变量生成时，通过 hook 触发：
   - voxel 特征采样
   - `concat_linear` 融合
6. `VolumePointStage1Model` 构造：
   - `atom_tokens`
   - `atom_hidden`
   - `atom_logits`
7. `VoxelPointStage1Wrapper` 计算：
   - atom 主损失
   - voxel aux 辅助损失
8. `train.py` 负责：
   - DataModule
   - DDP
   - logger
   - checkpoint
   - optimizer/scheduler 注入

当前 `forward()` 输出字典的完整键为：

- `atom_logits`
- `atom_hidden`
- `atom_tokens`
- `fused_point_feat`
- `sampled_point_fusion_feat_dict`
- `voxel_logits_aux`
- `voxel_outputs`
- `point_outputs`
- `embed_output`（仅在 embed head 启用时才有值）
- `voxel_recycle_out`
- `point_recycle_out`
- `recycle_passes_used`

## 2. 当前损失定义

当前损失配置来自 `configs/loss/focal_default.yaml`：

- `atom_loss = BinaryFocalLossWithAlpha`
- `voxel_aux_loss = BinaryFocalLossWithAlpha`
- `atom_loss_weight = 1.0`
- `voxel_aux_loss_weight = 0.2`

当前两个损失的参数为：

- `gamma = 2.0`
- `scale = 1.0`（两者均为 1.0）
- `alpha_tune = [0.75, 0.25]`
- `use_adaptive_alpha = false`

### 2.1 atom loss

当前调用方式是：

- `outputs["atom_logits"]`
- `batch["atom_label"]`
- `hardmask=batch["atom_valid_mask"]`
- `reduction="mean"`

因此当前 atom loss 的真实监督范围由：

- `atom_valid_mask`

决定。

### 2.2 voxel aux loss

当前调用方式是：

- `outputs["voxel_logits_aux"]`
- `batch["voxel_label"]`
- `hardmask = hardmask.bool() & voxel_valid_mask.bool()`
- `reduction="mean"`

因此当前 voxel aux loss 的真实监督范围由：

1. `hardmask`
2. `voxel_valid_mask`

共同决定。

在当前 `fused` 主配置下：

- `valid_crop_margin = 0`

所以 `voxel_valid_mask` 全 True，当前实际有效范围主要由 `hardmask` 决定。
